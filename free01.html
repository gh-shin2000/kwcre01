如何进行免费的网页托管或静态网站发布

要进行免费的网页托管或静态网站发布，你可以使用多个成熟、稳定且完全免费的平台。这些平台支持 HTML、CSS、JavaScript 等静态资源，适合个人主页、作品集、博客、项目演示等场景。

以下是目前（截至 2026 年）最推荐的 免费静态网站托管服务：

✅ 1. GitHub Pages（最经典、可靠）

  • 网址：https://pages.github.com

  • 特点：

    • 完全免费，无流量限制（合理使用范围内）
    • 支持自定义域名（可绑定自己的 .com 域名）
    • 与 GitHub 仓库无缝集成，通过推送代码自动部署
    • 支持 Jekyll 静态博客（也可用其他生成器如 Hugo、VuePress）

  • 限制：

    • 仅支持静态内容（不能运行 PHP、Node.js 后端）
    • 仓库需公开（私有仓库的 GitHub Pages 在免费账户下不可用）

  • 适合人群：开发者、学生、技术博客作者

📌 示例网址：https://yourusername.github.io

✅ 2. Vercel（前端开发者首选）

  • 网址：https://vercel.com

  • 特点：

    • 免费计划非常慷慨（每月 100GB 带宽）
    • 支持 Git 自动部署（连接 GitHub/GitLab/Bitbucket）
    • 一键部署 React、Vue、Svelte、Next.js 等现代框架
    • 自动 HTTPS + 全球 CDN 加速
    • 支持自定义域名

  • 限制：

    • Serverless 函数有执行时间限制（免费版 10 秒）
    • 但纯静态网站无任何问题

  • 适合人群：前端开发者、Web 项目展示

📌 部署方式：把代码推送到 GitHub 仓库 → 导入 Vercel → 自动上线

✅ 3. Netlify（功能强大，体验优秀）

  • 网址：https://www.netlify.com

  • 特点：

    • 免费计划包含自动部署、CDN、HTTPS、表单处理、重定向等
    • 拖拽上传文件即可发布（无需 Git）
    • 支持预览部署（Pull Request 自动生成预览链接）
    • 可绑定自定义域名

  • 限制：

    • 带宽和构建时间有免费额度（对个人项目足够）

  • 适合人群：设计师、非程序员、快速原型发布者

📌 支持直接上传 ZIP 或连接 Git 仓库

✅ 4. Cloudflare Pages（性能极佳）

  • 网址：https://pages.cloudflare.com

  • 特点：

    • 免费、无限带宽、全球边缘网络加速
    • 与 Cloudflare 生态深度集成（安全、DNS、缓存）
    • 支持 Git 自动部署和预览
    • 构建速度快

  • 适合人群：追求极致加载速度和安全性的用户

✅ 5. GitLab Pages

  • 类似 GitHub Pages，适用于使用 GitLab 的用户。
  • 支持私有项目免费发布（比 GitHub 更宽松）。
  • 配置稍复杂，需写 .gitlab-ci.yml 文件。

🚀 快速上手示例（以 GitHub Pages 为例）：

  1. 注册 GitHub 账号
  2. 新建一个仓库，命名为：你的用户名.github.io（例如 alice.github.io）
  3. 将你的网页文件（index.html, style.css 等）上传到该仓库
  4. 等待 1 分钟，访问 https://你的用户名.github.io 即可看到网站！

💡 提示：即使不会 Git，也可以直接在 GitHub 网页上点 “Add file” 上传。

⚠️ 注意事项：

  • 所有上述服务仅支持静态网站（不能运行数据库、PHP、Python 后端等）。
  • 如需后端功能，可结合 Firebase、Supabase、Railway、Render 等免费 BaaS/PaaS 服务。
  • 敏感信息（如 API 密钥）不要硬编码在前端代码中。

✅ 总结推荐：

所有选项均完全免费，无需信用卡，注册即用。

为什么我的网站部署后几分钟后还是404？

部署后出现 404 错误（“There isn't a GitHub Pages site here”）是非常常见的情况。既然你已经按照规范创建了仓库，那么问题通常出在文件缺失、分支设置或路径配置上。

别着急，请按照以下步骤逐一排查，通常能解决 99% 的问题：

🔍 1. 检查“入口文件”是否存在 (最常见原因)

GitHub Pages 需要一个入口文件才能运行。请确认你的仓库根目录下（或者你指定的 docs 文件夹内）有以下文件之一：

  • index.html
  • index.md
  • README.md[[source_group_web_1]]

注意：

  • 文件名必须是小写的 index.html，不能是 Index.html 或 index.HTML（GitHub 服务器区分大小写）[[source_group_web_2]]。
  • 如果你上传的是前端框架（如 Vue/React/Vite）的源代码（包含 .vue 或 .jsx 文件），而没有上传 dist 文件夹里的构建后的静态文件，GitHub Pages 是无法直接运行的，必须先构建（Build）生成 index.html[[source_group_web_3]]。

⚙️ 2. 检查 Settings 中的源（Source）设置

有时候仓库里有文件，但 GitHub Pages 没有去读取它们。

  1. 进入仓库的 Settings -> Pages。

  2. 查看

    Branch

    （分支）设置：

    • 如果你的文件在 main 分支上，请确保这里选的是 main[[source_group_web_4]]。
    • 如果你的文件在 gh-pages 分支上，请确保这里选的是 gh-pages。

  3. 查看

    Folder

    （文件夹）设置：

    • 如果你的 index.html 在根目录，选择 / (root)。
    • 如果你的文件都在 docs 文件夹里，请选择 /docs[[source_group_web_5]]。

  4. 点击 Save 保存设置（即使没改也要点一下，有时候能触发重新部署）。

📂 3. 检查仓库名是否与“路径”冲突 (针对项目站点)

这一点很容易被忽视。如果你没有使用 用户名.github.io 这种仓库名（即你是在用项目站点），那么你的网页资源路径必须匹配仓库名。

  • 情况：你的仓库名是 my-blog。
  • 错误配置：在 vite.config.js 或 vue.config.js 中，base 或 publicPath 设置为了 '/'。
  • 结果：页面会尝试去 你的用户名.github.io/index.js 找资源，但实际上资源在 你的用户名.github.io/my-blog/index.js，导致 404。
  • 解决：需要将配置改为 '/my-blog/'[[source_group_web_6]]。

🛠️ 4. 检查是否缺少 .nojekyll 文件

如果你的网站包含下划线开头的文件夹（例如 _assets），GitHub 默认使用的 Jekyll 引擎会忽略这些文件夹，导致资源 404。

  • 解决：在项目根目录（或 public 目录）下创建一个空文件，命名为 .nojekyll，然后重新提交代码[[source_group_web_7]]。

⏳ 5. 耐心等待与缓存问题

  • 部署延迟：虽然通常很快，但 GitHub Pages 的全球 CDN 缓存生效可能需要 几分钟到一小时[[source_group_web_8]]。如果刚 Push 完代码，可以稍等片刻再试。
  • 浏览器缓存：如果你之前访问过该地址且是 404，浏览器可能会缓存这个错误页面。尝试使用 无痕模式 或者清除浏览器缓存后刷新[[source_group_web_9]]。

📋 快速排查清单

建议你先去 Settings -> Pages 看一眼当前的部署状态，通常那里会直接提示你“找不到 index 文件”或者“分支错误”。
